# AP Health IQ — Fine-Tuned Model Card

## Model

- **Name:** `ap-health` (Llama 3.2 3B Instruct, fine-tuned on AP clinical data)
- **Base model:** `unsloth/Llama-3.2-3B-Instruct`
- **Method:** LoRA (rank 16, alpha 16, dropout 0)
- **Target modules:** q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj
- **Trainable params:** 24.3M of 3.24B (0.75%)
- **Quantization:** Q4_K_M GGUF (~2 GB final size)
- **License inheritance:** Llama 3.2 Community License (Meta)

## Training Data

> **Provenance guarantee — training data is real-only.**
> The fine-tuned model was trained exclusively on the **10,045 real anonymised AP government OPD records** from the source Excel dataset.
> The ~91,000 synthetic records that exist in the live demo database (flagged `is_synthetic=True` for dashboard scale-testing) were **never** used in training. The training pipeline (`backend/ai/prepare_training_data.py`) reads directly from the Excel file via `pd.read_excel(DATASET_PATH, ...)` and does not touch the SQLite database. Reviewers can verify this in 30 seconds by grepping the file for `read_excel` / `read_sql` — only `read_excel` appears.
> Rationale: training on synthetic data risks pattern leakage, silent overfitting to generator biases, and reduces the credibility of the model for real-world clinical decision support.

- **Source:** Real AP government OPD records (Excel dataset, 10,045 rows)
- **Pairs generated:** 15,206 instruction-input-output triples
- **Pair types:**
  - Per-record disease classification (ICD-10 + SNOMED + category + action)
  - BP / RBS vitals interpretation with risk stratification
  - District-level outbreak trend pairs (case count vs baseline → severity + actions)
  - Hand-crafted AP-specific Q&A (top diseases, highest-burden district, etc.)
- **Generated by:** `backend/ai/prepare_training_data.py`
- **File:** `backend/ai/training_data.jsonl` (6.1 MB)
- **Schema:** `{instruction, input, output}` (Alpaca format) — no nested fields, no metadata.

### Limitations and intended use

- **Intended for clinical decision support, not diagnosis.** Outputs must be reviewed by a qualified Medical Officer before any treatment decision.
- **AP-context-specific.** Trained only on AP OPD distributions — may underperform on diseases or demographics not represented in AP data (e.g., tropical diseases prevalent in NE India).
- **Not for individual patient identification.** Even though training data is anonymised, the model should not be queried with directly-identifying patient data; pass only complaint + vitals + district in production prompts.
- **Templated outputs.** Final loss of 0.09 indicates the model learned the structured Alpaca template very well; may be over-confident on out-of-distribution queries.

## Training Run

- **Hardware:** Google Colab T4 GPU (free tier)
- **Framework:** Unsloth (2× faster LoRA on consumer GPUs)
- **Hyperparameters:**
  - Batch size: 2 per device × 4 grad accumulation = 8 effective
  - Learning rate: 2e-4, linear schedule, 5 warmup steps
  - Optimizer: AdamW 8-bit
  - Mixed precision: bf16 (T4 supported)
  - Epochs: 3 planned, **early-stopped at step 416** (~7% of run) once loss converged at ~0.09
- **Final training loss:** ~0.09 (excellent — task is templated, model converged fast)

## Production Deployment Status

- **Currently in production:** No — Groq Cloud (`llama-3.1-8b-instant`) is used as the production AI provider for reliability and zero-cost serverless inference.
- **Reason:** Demo bulletproofing. The fine-tuned GGUF is included as a deployable artifact and methodology proof.
- **To activate the fine-tuned model:**
  1. Spin up Ollama on a GPU host (RunPod RTX 4090 ~₹30/hr, or local laptop with NVIDIA GPU)
  2. Load the GGUF: `cp ap-health-unsloth.Q4_K_M.gguf ap-health-q4.gguf && ollama create ap-health -f Modelfile`
  3. Set Railway env vars:
     ```
     AI_PROVIDER=ollama
     OLLAMA_BASE_URL=https://<your-ollama-host>:11434
     OLLAMA_MODEL=ap-health:latest
     ```
  4. No code changes needed — `backend/services/ollama_service.py` already supports both providers.
